# Definition of Done (DoD) Checklist

> **Purpose**: Criteria that a ticket must meet BEFORE being considered complete and ready for QA.
> **Evaluated by**: Hook `dtc-write-guard` (PreToolUse: Write|Edit) and skill `pr-description`
> **Associated Gate**: Gate 4 — Code Quality
> **Trigger**: Evaluated automatically before each `git push` and when moving ticket to "Ready for QA"

---

## 1. Code Criteria

### 1.1 Code Review

- [ ] **PR Created** — Pull Request exists in GitHub with feature branch → develop/main
- [ ] **PR Description Complete** — Generated by skill `pr-description`: what, why, how to test, breaking changes
- [ ] **Peer Review Approved** — At least 1 peer developer has approved the PR
- [ ] **Tech Lead Review** — Tech Lead has approved (or explicitly delegated)
- [ ] **Comments Resolved** — 0 open/pending conversations in the PR
- [ ] **No Self-Merge** — PR author is not the one merging it

### 1.2 Code Quality

- [ ] **Unit Tests Pass** — 100% green in pipeline, coverage ≥80% in business logic
- [ ] **SAST/SCA Clean** — 0 Critical or High vulnerabilities (SonarQube + Dependabot/Snyk)
- [ ] **Lint/Format Clean** — 0 ESLint warnings, 0 Prettier errors, 0 TypeScript errors
- [ ] **Build Successful** — Complete CI pipeline without errors or ignored warnings
- [ ] **No Orphan TODOs** — New `TODO`s have linked Jira ticket: `// TODO [PROJ-123]: description`
- [ ] **No console.log** — No `console.log`, `console.debug` outside logging utilities
- [ ] **No Hardcoded Secrets** — No API keys, passwords, tokens in code (verified by SAST)

### 1.3 Tests

- [ ] **Tests for New Code** — If business logic added, tests exist that cover it
- [ ] **BDD Criteria Verified** — Dev has manually verified Given/When/Then scenarios of ticket
- [ ] **No Known Regressions** — Relevant regression tests executed and passing
- [ ] **Acceptable Performance** — No measurable degradation in affected endpoints (verify with APM tools)

---

## 2. Documentation Criteria

> **DTC Rule (Docs Travel With Code)**: The Dev who changes code updates affected docs IN THE SAME PR.

- [ ] **PR Description Complete** — Generated by skill `pr-description` with: what changed, why, how to test
- [ ] **ADR if Architectural Decision** — If change introduces new pattern or chooses between alternatives → ADR created
- [ ] **Affected Docs Updated IN THIS PR** — If change affects DB/API/components/deps/pipeline → corresponding docs updated in same PR
- [ ] **Inline Docs for Complex Code** — Complex logic documented with clear comments
- [ ] **Tech Debt Registered** — If technical debt identified during implementation → registered with skill `tech-debt`
- [ ] **Changelog Entry** — If feature/fix visible → entry in changelog or conventional commits
- [ ] **Ecosystem Docs Synchronized** — If skill, command, rule, checklist or template created/modified → counts updated in all sources of truth

---

## 3. Delivery Criteria (Dev→QA Handoff)

- [ ] **Dev→QA Handoff Generated** — Document generated with skill `dev-handoff-qa` attached to ticket
- [ ] **Handoff Content Complete** — Includes: what implemented, relevant changes (endpoints, DB, config), how to test
- [ ] **Jira Status Updated** — Ticket moved to "Ready for QA"
- [ ] **QA Notified** — Notification sent via Slack with ticket link and handoff
- [ ] **Environment Deployed** — Code available in staging environment for QA testing
- [ ] **Test Data Available** — Data needed for testing is in environment or documented in handoff

---

## 4. Severity Levels

| Level     | Meaning                         | Action                                                    |
| --------- | ------------------------------- | --------------------------------------------------------- |
| **BLOCK** | Critical criterion not met      | Push rejected, dev must resolve before retrying           |
| **WARN**  | Important criterion unfulfilled | Push allowed, but warning recorded and Tech Lead notified |
| **INFO**  | Desirable criterion unfulfilled | Push allowed, information recorded for retrospective      |

---

## 5. Emergency Override

In exceptional cases (critical hotfix in production), Tech Lead can override:

```bash
git push --force-with-lease -o skip-hooks
```

**Conditions for override:**

1. Hotfix documented in #incidents channel
2. Tech Lead explicitly approves (Slack timestamp as evidence)
3. Jira ticket of type "Hotfix" created with incident link
4. DoD debt registered as mandatory follow-up task

---

## 6. Criteria Requiring Human Judgment

AI cannot evaluate these criteria — they require context and experience:

| Criterion                      | Who Evaluates | What to Look For                                                       |
| ------------------------------ | ------------- | ---------------------------------------------------------------------- |
| **Code Review Quality**        | Tech Lead     | Not just "approved" — reviewer actually reviewed and understood change |
| **Documentation Completeness** | Tech Lead     | Is it sufficient for someone new to understand the change?             |
| **Acceptable Performance**     | Dev + DevOps  | Pre/post change benchmarks, not just "seems fast"                      |
| **Regression Impact**          | Dev + QA      | Can the change break something unexpected outside ticket scope?        |
| **Clean Design**               | Tech Lead     | Code not only works — it's maintainable, testable, extensible          |

---

## 7. Anti-patterns

| Anti-pattern                 | Problem                                      | Solution                                              |
| ---------------------------- | -------------------------------------------- | ----------------------------------------------------- |
| "CI is green" = "It's ready" | CI passes but no new tests                   | Verify tests for new code                             |
| "Approved without reading"   | Reviewer approves without actually reviewing | Tech Lead audits reviews randomly                     |
| "Will document later"        | Missing handoff, QA has no context           | Hook blocks transition without handoff attached       |
| "TODOs are temporary"        | TODOs without tickets stay forever           | Hook verifies `TODO [PROJ-XXX]` format                |
| "Override because urgent"    | Override abuse to skip DoD                   | Override requires Hotfix ticket + mandatory follow-up |
| "Performance measured later" | Degradation detected in production           | Dev measures before/after in staging                  |

---

## 8. Connection with SDLC Flow

```
Sprint Planning (Gate 3) → Development → DoD evaluated (hook dtc-write-guard)
    ↓                                          ↓
If PASS → Ready for QA → QA Testing (Gate 5)
If FAIL → Dev fixes → Re-evaluate DoD
If OVERRIDE → Hotfix → DoD debt registered → Follow-up sprint
```
