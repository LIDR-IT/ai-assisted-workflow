# CHANGELOG.md Template

> **Purpose**: Base template to initialize the CHANGELOG.md of any project.
> **Format**: [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) + [Conventional Commits](https://www.conventionalcommits.org/).
> **Generated by**: `/update-changelog [version]` (automatic from merged PRs).
> **Complements**: Skill `release-notes`, which generates a two-level changelog (business + technical).

---

## Base Format

```markdown
# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- {New features}

### Changed

- {Changes to existing features}

### Deprecated

- {Features that will be removed}

### Removed

- {Removed features}

### Fixed

- {Bug fixes}

### Security

- {Vulnerability fixes}

## [1.0.0] - YYYY-MM-DD

### Added

- Initial project release
- {Feature 1}
- {Feature 2}

[Unreleased]: https://github.com/{org}/{repo}/compare/v1.0.0...HEAD
[1.0.0]: https://github.com/{org}/{repo}/releases/tag/v1.0.0
```

## Changelog Rules

| Rule                            | Detail                                              |
| ------------------------------- | --------------------------------------------------- |
| **Audience**                    | Humans, not machines. Clear and direct language     |
| **One entry per version**       | Grouped by change type (Added, Changed, etc.)       |
| **Unreleased always visible**   | Section at the top for changes pending release      |
| **Comparison links**            | Footer with {{VCS_TOOL}} compare links between tags |
| **Reverse chronological order** | Most recent versions at the top                     |
| **ISO 8601 date**               | YYYY-MM-DD format                                   |
| **Strict semver**               | MAJOR.MINOR.PATCH according to impact               |

## Conventional Commits → Changelog Mapping

| Commit type         | Changelog Section                  |
| ------------------- | ---------------------------------- |
| `feat`              | Added                              |
| `fix`               | Fixed                              |
| `docs`              | Changed (if it affects public API) |
| `refactor`          | Changed                            |
| `perf`              | Changed                            |
| `BREAKING CHANGE`   | Changed (with a prominent note)    |
| `deprecate`         | Deprecated                         |
| `security` / `vuln` | Security                           |

## Update Flow

```
1. PRs are merged to develop with conventional commits
2. Before release: /create-release-notes [version]
   → Skill release-notes generates the two-level changelog
3. /update-changelog [version]
   → Reads PRs merged since the last tag
   → Classifies by type (feat, fix, etc.)
   → Generates the entry in CHANGELOG.md
   → Commit + tag vX.Y.Z
4. Gate 7 validates that CHANGELOG.md is up to date
```
