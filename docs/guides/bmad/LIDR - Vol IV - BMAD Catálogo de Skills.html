0Bienvenida

Este libro es la referencia comando a comando del método. No está pensado para leerse de principio a fin: se consulta cuando vas a invocar una skill concreta y necesitas saber qué espera, qué produce y cuánto tarda.

El método incluye 66 skills en total. Este catálogo cubre las 32 que componen el flujo del proyecto (módulos BMM y TEA) con detalle pleno, y deja las otras 34 como apéndice resumido (módulos CIS, BMB, Core, agentes con persona y skills auxiliares de BMM).

0.2¿Qué es una skill?

Una skill es un workflow ejecutable que sigue una serie de pasos estructurados para producir un artefacto concreto (un documento, un test, una decisión, una refactorización). Se invoca con un /comando en el agente de IA.

En el sistema instalado, cada skill vive en .claude/skills/<nombre-de-la-skill>/. Dentro hay un archivo principal (SKILL.md) que orquesta la ejecución y, en muchas skills, una carpeta de sub‑pasos. La carpeta _bmad/ contiene la configuración del método (manifests, módulos), no las skills en sí.

Modos CEV en las skills TEA Las nueve skills TEA usan un patrón uniforme con tres modos: Create (genera el artefacto desde cero, en steps-c/), Edit (modifica un artefacto existente, en steps-e/) y Validate (audita el artefacto sin tocarlo, en steps-v/). El modo se selecciona automáticamente según el contexto, o se fuerza con un argumento.

0.3Anatomía de una ficha

Cada ficha del Sec. 2 sigue siempre el mismo orden, para que el lector localice la información en el mismo sitio sin tener que leerla entera:

  1. Cabecera — comando, módulo, fase y obligatoriedad.
  2. Resuelve — el problema que aborda en una sola frase.
  3. Cuándo invocarla — tres situaciones concretas.
  4. Necesita antes — prerrequisitos verificables.
  5. Produce — artefacto de salida y dónde se guarda.
  6. Cómo funciona — sub‑pasos o fases internas.
  7. Cuánto tarda — rango orientativo.
  8. Consultar también — skills relacionadas (cuando aplica).

2.0Fase 0 — Aprendizaje (opcional, solo si se adopta TEA)

01 /bmad-teach-me-testing Test Architect OPC Fase 0
Resuelve: Forma al equipo en testing antes de adoptar TEA mediante un programa autodirigido con persistencia de progreso entre sesiones.
Cuándo invocarla
  • El equipo adopta TEA por primera vez.
  • Onboarding de Dev o QA al stack de testing.
  • Antes de /bmad-testarch-test-design si los conceptos base no están claros.
Necesita antes
  • Configuración del módulo TEA (idioma, nombre del proyecto).
  • No depende de PRD ni arquitectura.
Produce
Notas de cada sesión + archivo de progreso reanudable. Al final, certificado de finalización del programa.
Cómo funciona
12 sub‑pasos en modo Create, de los que 7 son las sesiones académicas (step-04-session-01 a session-07). Incluye modos Validate y Edit para auditar o modificar el progreso.
Cuánto tarda
2–4 h por sesión × 7 sesiones = entre 14 y 28 h, distribuidas en una o dos semanas.
Consultar también
Libro 5 (Protocolo TEA) para la estrategia de testing en proyecto real.

03 /bmad-product-brief BMad Method REC Fase 1
Resuelve: Produce un brief de producto persuasivo de una a dos páginas, listo para alimentar el PRD, mediante descubrimiento colaborativo y análisis de artefactos previos.
Cuándo invocarla
  • Tras brainstorm y research, antes del PRD.
  • Para alinear stakeholders en una página.
  • Para refrescar un brief antiguo sin reescribirlo desde cero.
Necesita antes
  • Idea inicial o brief existente.
  • Opcional: outputs previos de brainstorming, market o domain research.
Produce
Brief ejecutivo en la carpeta de planificación. Opcionalmente, un resumen comprimido optimizado para alimentar al LLM al crear el PRD.
Cómo funciona
Cinco etapas (entender intención, descubrimiento contextual, elicitación guiada, redacción y revisión, finalización). Tres modos: guiado, rápido y autónomo.
Cuánto tarda
Guiado: 45–90 min. Rápido y autónomo: ~30 min.
Consultar también
/bmad-prfaq (apéndice) como alternativa cuando la idea aún no está validada.

2.2Fase 2 — Planificación

07 /bmad-create-prd BMad Method REQ Fase 2
Resuelve: Produce el PRDPRD: Product Requirements Document. Documento que describe lo que el producto debe hacer (requisitos funcionales) y cómo de bien debe hacerlo (requisitos no funcionales). Es el contrato entre producto y equipo técnico. exhaustivo del producto desde cero, con seguimiento de progreso entre sesiones.
Cuándo invocarla
  • Greenfield: primer PRD del proyecto.
  • Brownfield: épica grande con alcance independiente.
  • Refresh completo de un PRD obsoleto.
Necesita antes
  • Product Brief.
  • Recomendado: market y domain research.
Produce
{planning}/prd.md. Incluye seguimiento de pasos completados en el frontmatter para reanudar.
Cómo funciona
15 sub‑pasos en steps-c/: init, continue, discovery, vision, executive summary, success criteria, journeys, domain, innovation, project type, scoping, FRs, NFRs, polish, complete.
Cuánto tarda
3–5 horas distribuidas en dos o tres sesiones.
Consultar también
/bmad-validate-prd al terminar; /bmad-edit-prd para correcciones posteriores.
08 /bmad-validate-prd BMad Method REC Fase 2
Resuelve: Audita el PRD contra trece dimensiones de calidad y produce un reporte de incidencias priorizadas.
Cuándo invocarla
  • Tras /bmad-create-prd, antes de arquitectura o UX.
  • Revisión de un PRD escrito por otra persona.
  • Antes de /bmad-check-implementation-readiness.
Necesita antes
  • Path al PRD a validar.
Produce
Reporte en formato markdown con incidencias categorizadas por dimensión y recomendaciones priorizadas.
Cómo funciona
14 sub‑pasos de validación en steps-v/: descubrimiento, formato, paridad, densidad, cobertura del brief, medibilidad, trazabilidad, intrusión de implementación, dominio, tipo de proyecto, heurísticas, calidad holística, completitud, reporte.
Cuánto tarda
60–90 minutos.
Consultar también
/bmad-edit-prd para resolver las incidencias detectadas.
Recomendación LIDR El método marca /bmad-validate-prd como opcional. Recomendamos tratarla como recomendada porque saltarla es la causa más frecuente de retrabajo en Fase 4: los huecos del PRD viajan al backlog y se hacen visibles cuando el equipo ya está implementando. Para ganar objetividad, conviene ejecutarla con un LLM distinto al que produjo el PRD.

Cuándo invocarla

  • Tras PRD aprobado, antes de la arquitectura.

Produce
{planning}/ux-design-specification.md basado en plantilla.

15 sub‑pasos: init, continue, discovery, experiencia central, respuesta emocional, inspiración, sistema de diseño, experiencia que define al producto, fundamentos visuales, direcciones de diseño, viajes de usuario, estrategia de componentes, patrones de UX, responsivo y accesibilidad, completar.

4–6 horas distribuidas.
Consultar también
Libro 3 (Plan de Sesiones) para encajarla en la sesión correspondiente.
10 /bmad-testarch-nfr Test Architect REC Fase 2 & 4
Resuelve: Evalúa los requisitos no funcionales (rendimiento, seguridad, fiabilidad, escalabilidad) con validación basada en evidencia y desenlace determinista.
Cuándo invocarla
  • En Fase 2 (modo setup) para fijar umbrales en el PRD.
  • Mid‑ciclo (modo check) para detectar desviaciones.
  • Pre‑release (modo gate) como decisión final.
Necesita antes
  • PRD con NFRNFR: Non-Functional Requirement. Requisito que describe cómo de bien debe comportarse el sistema (rendimiento, seguridad, fiabilidad, escalabilidad), en contraposición a los functional requirements que describen qué debe hacer. identificados.
  • Para check y gate: evidencia (load tests, security scans, métricas de fiabilidad).
Produce
Reporte por categoría con scoring, evidencia recogida y recomendaciones. Modo gate emite firma final: PASS, CONCERNS, WAIVED o FAIL.
Cómo funciona
Modo Create: 11 sub‑pasos con sub‑agentes paralelos por categoría (security, performance, reliability, scalability). Modos Validate y Edit: dos pasos cada uno.
Cuánto tarda
50–75 minutos por invocación.
Consultar también
Libro 5 (Protocolo TEA) para el detalle de cada categoría.
Visión LIDR En el método, NFR es una sola skill con tres modos. Nosotros la pensamos como tres momentos del proyecto — setup en Fase 2, check a mitad del ciclo, gate antes del release — porque cada momento responde una pregunta distinta: setup fija el contrato, check vigila la deriva, gate firma la salida. El catálogo solo lista la skill una vez; las invocaciones son tres.

  • PRD completado.

Produce
{planning}/architecture.md con decisiones técnicas, patrones y restricciones.

9 sub‑pasos: init, continue, contexto, starter, decisiones, patrones, estructura, validación, complete.

60–90 minutos.

12 /bmad-testarch-test-design Test Architect REQ TEA Fase 3
Resuelve: Plan de testing basado en riesgo con clasificación de prioridades (P0–P3), criterios de entrada y salida y orden de ejecución.
Cuándo invocarla
  • Pre‑gate de la fase de solución, para validar testabilidad arquitectónica.
  • Por cada épica grande en Fase 4 (modo epic‑level).
  • Para fijar criterios de entrada y salida con QA.
Necesita antes
  • System‑level: arquitectura y ADRADR: Architecture Decision Record. Documento corto que captura una decisión técnica (qué se eligió, qué se descartó y por qué), con sus trade-offs. Es el rastro auditable de las decisiones de arquitectura, distinto del documento general de arquitectura..
  • Epic‑level: épica + historias + PRD.
Produce
Plan de testing y matriz de riesgos. Plantillas: system, epic, qa, handoff y architecture.
Cómo funciona
Modo Create con 6 sub‑pasos: detect mode, load context, riesgos y testabilidad, plan de cobertura, generación de salida, resume. Auto‑detecta si está en system‑level o epic‑level.
Cuánto tarda
30–45 minutos.
Consultar también
/bmad-testarch-framework después, en cadena estricta.
13 /bmad-testarch-framework Test Architect REQ TEA Fase 3
Resuelve: Andamiaje del framework de tests listo para producción (Playwright o Cypress) con fixtures, helpers, configuración y buenas prácticas.
Cuándo invocarla
  • Inicializar testing en proyecto nuevo o repositorio sin tests.
  • Migración entre frameworks.
  • Tests sueltos sin estructura coherente.
Necesita antes
  • Test Design ejecutado.
  • Stack tecnológico decidido.
  • Acceso de escritura al repositorio.
Produce
playwright.config.ts o cypress.config.ts; fixtures, helpers, page objects, scripts; README de setup.
Cómo funciona
Modo Create con 6 sub‑pasos: preflight, selección de framework, scaffolding, docs y scripts, validación, resume. Configurable por preferencia, tamaño del proyecto y uso de TypeScript.
Cuánto tarda
25–35 minutos.
Consultar también
/bmad-testarch-ci después, para enlazar la pipeline.
14 /bmad-testarch-ci Test Architect REQ TEA Fase 3

16 /bmad-check-implementation-readiness BMad Method REQ Fase 3

Reporte de readiness con huecos detectados, severidad y recomendaciones de resolución.
Cómo funciona
6 sub‑pasos: descubrimiento de documentos, análisis del PRD, validación de cobertura por épica, alineación con UX, revisión de calidad de épicas, evaluación final.
Cuánto tarda
60–90 minutos.
Consultar también

18 /bmad-create-story BMad Method REQ Fase 4 · ciclo
Resuelve: Crea el archivo de la historia con todo el contexto que el agente Dev necesita: análisis del PRD, arquitectura, UX, historias previas, historial de Git e investigación web actualizada.
Cuándo invocarla
  • Antes de cada /bmad-dev-story.
  • Para preparar la siguiente historia del backlog.
  • Para crear una historia específica solicitada por el usuario.
Necesita antes
  • epics.md, prd.md, architecture.md, UX si aplica.
  • sprint-status.yaml.
  • Acceso al historial de Git.
Produce
{implementation}/{story-key}.md con requisitos, criterios de aceptación, tareas, notas técnicas, inteligencia de Git, investigación web y referencia a project-context.md.
Cómo funciona
Dos acciones: create (REQ) genera la siguiente historia y validate (REC) verifica su completitud antes del desarrollo. Seis fases: determinar objetivo, cargar artefactos, análisis arquitectónico, investigación web, creación del archivo, actualización del sprint‑status. Sin carpeta de sub‑pasos.
Cuánto tarda
40–60 minutos.
Consultar también
/bmad-testarch-atdd después, en cadena estricta.
19 /bmad-testarch-atdd Test Architect REQ TEA Fase 4 · ciclo
Resuelve: Genera tests de aceptación que fallan (fase roja del ciclo red‑green‑refactorred‑green‑refactor: ciclo de Test‑Driven Development en tres pasos. Rojo: escribir un test que falle. Verde: implementar lo mínimo para que el test pase. Refactor: limpiar el código manteniendo el test verde.) antes de la implementación, en niveles apropiados (E2E, API, componente).
Cuándo invocarla
  • Tras /bmad-create-story, antes de /bmad-dev-story.
  • Cuando el equipo quiere ATDD/TDD estricto.
  • Para convertir criterios de aceptación en tests ejecutables.
Necesita antes
  • Historia con criterios de aceptación y requisitos técnicos.
  • Framework configurado (/bmad-testarch-framework).
Produce
Tests fallidos por nivel + fixtures + helpers; checklist de implementación; trazabilidad criterios↔tests.
Cómo funciona
Modo Create con 9 sub‑pasos y sub‑agentes paralelos por nivel (API, E2E). Modos Validate y Edit disponibles.
Cuánto tarda
40–60 minutos.
Consultar también
/bmad-dev-story implementa lo que estos tests piden (fase verde).
20 /bmad-dev-story BMad Method REQ Fase 4 · ciclo
Resuelve: Implementa la historia siguiendo su archivo de especificación con el ciclo red‑green‑refactor: tareas, tests, validaciones y marca la historia como review.
Cuándo invocarla
  • Historia en estado ready‑for‑dev.
  • Continuación tras code review (cuando vuelve con cambios).
  • Tras tests ATDD en rojo.
Necesita antes
  • Archivo de historia con tareas, criterios y notas.
  • sprint-status.yaml.
  • Repositorio Git en estado limpio o committable.
Produce
Código nuevo o modificado, tests unitarios y de integración, archivo de historia actualizado (tareas marcadas, change log, lista de archivos, estado review) y sprint‑status actualizado.
Cómo funciona
Diez fases: localizar la siguiente historia lista, cargar contexto, detectar continuación tras review, marcar en progreso, implementar red‑green‑refactor, autorar tests, ejecutar validaciones, marcar completa, finalizar, comunicar. Sin carpeta de sub‑pasos.
Cuánto tarda
120–180 minutos por historia (variable según complejidad).
Consultar también
/bmad-testarch-automate después; /bmad-code-review antes del merge.
21 /bmad-testarch-automate Test Architect REC Fase 4 · ciclo
Resuelve: Expansión de la cobertura tras la implementación: genera tests priorizados a nivel E2E, API, componente y unitario usando sub‑agentes paralelos.
Cuándo invocarla
  • Tras /bmad-dev-story y /bmad-code-review, para ampliar más allá de los criterios de aceptación.
  • En proyectos brownfield que necesitan cobertura retroactiva.
  • Para alcanzar un umbral de cobertura antes del release.
Necesita antes
  • Codebase existente.
  • Modo BMad-Integrated: historia, PRD y test design. Modo Standalone: solo código.
Produce
Tests por nivel + reporte de cobertura (actual contra objetivo); fixtures, helpers y cola de implementación.
Cómo funciona
Modo Create con 9 sub‑pasos: preflight, identificar objetivos, generar tests, sub‑agentes API, backend y E2E, agregar, validar, resume.
Cuánto tarda
45–60 minutos.
Consultar también
/bmad-testarch-test-review para auditar la calidad de los tests acumulados.
22 /bmad-code-review BMad Method REC Fase 4 · ciclo
Resuelve: Revisión adversarial pre‑merge usando tres capas paralelas: Blind Hunter (busca bugs sin contexto), Edge Case Hunter (recorre ramas y condiciones límite) y Acceptance Auditor (valida cumplimiento de criterios de aceptación).
Cuándo invocarla
  • Tras /bmad-dev-story con la historia en review.
  • Antes de mergear a la rama principal.
  • Cuando hay dudas sobre la calidad del cambio.
Necesita antes
  • Archivo de historia con lista de archivos modificados y criterios de aceptación.
  • Código implementado.
Produce
Sección "Senior Developer Review (AI)" añadida al archivo de historia, con triage en categorías de severidad (alta, media, baja) y lista de seguimiento accionable.
Cómo funciona
4 sub‑pasos: recolectar contexto, revisar (las tres capas paralelas), triar, presentar.
Cuánto tarda
30–50 minutos.
Consultar también
Recomendamos usar un LLM distinto al que implementó para máxima objetividad.
Recomendación LIDR El método marca /bmad-code-review como opcional. Recomendamos tratarlo como recomendado del ciclo: el code review captura defectos antes del merge usando un LLM distinto al que implementó. Saltarlo abarata el ciclo a corto plazo y lo encarece a medio plazo, porque los defectos llegan a la pipeline o a producción.

  • Auditoría regulatoria.

Matriz de trazabilidad + reporte de cobertura + decisión de gate.

25–35 minutos.

25 /bmad-retrospective BMad Method OPC Fase 4 · cierre
Resuelve: Retrospectiva formal tras épica: extrae lecciones aprendidas, evalúa el éxito y prepara la próxima épica. Formato conversacional multi‑agente facilitado por el Scrum Master.
Cuándo invocarla

Produce

Cuatro fases: descubrimiento de la épica, facilitación en formato party mode, revisión, preparación de la próxima épica. Sin carpeta de sub‑pasos.
Cuánto tarda
45–75 minutos.
Consultar también
/bmad-correct-course si aparecen issues mayores en la retro.

27 /bmad-correct-course BMad Method OPC Anytime · mid‑sprint

Cómo funciona
Seis fases: inicializar navegación, ejecutar checklist de análisis, redactar propuestas, generar Sprint Change Proposal, finalizar y enrutar, completar.
Cuánto tarda
60–90 minutos.
Consultar también
/bmad-edit-prd y /bmad-create-story según las decisiones que salgan.
28 /bmad-sprint-status BMad Method OPC Anytime · consulta

29 /bmad-document-project BMad Method OPC Brownfield
Resuelve: Documenta proyectos brownfield para alimentar contexto a agentes IA. Router que delega en sub‑workflows: full‑scan (rápido / profundo / exhaustivo) o deep‑dive (revisión literal sin muestreo).

Router en SKILL.md + dos sub‑workflows (full‑scan, deep‑dive), cada uno con sus propias instrucciones.

3 sub‑pasos: descubrir, generar, completar.

31 /bmad-quick-dev BMad Method OPC Brownfield
Resuelve: Workflow unificado intent→código que produce un artefacto reviewable en cinco pasos, sin pasar por PRD, arquitectura ni historias.
Cuándo invocarla
  • Bug fix puntual, refactor pequeño, retoque de UI o UX.
  • Feature menor sin necesidad de PRD.
  • Brownfield sin artefactos del método.
Necesita antes
  • Intent inicial del usuario.
  • Opcional: project-context.md y archivos de memoria del agente.
Produce
{implementation}/spec-wip.md append‑only; código implementado y presentación final (resumen, tests, diff).
Cómo funciona
Cinco pasos numerados (clarify and route, plan, implement, review, present) más un oneshot agresivo. No tiene carpeta steps/: archivos planos en la raíz.

20–90 minutos según el alcance.

No usar para features grandes ni cambios arquitectónicos: ahí entra el flujo completo de la Fase 2.
32 /bmad-qa-generate-e2e-tests Test Architect OPC Brownfield
Resuelve: Generación rápida de tests E2E y de API para código ya implementado (cobertura retroactiva). Auto‑detecta el framework existente.

  • Código fuente existente.

{implementation}/tests/test-{feature}.spec.ts ya validados; test-summary.md con cobertura y próximos pasos.
Cómo funciona
Cinco fases: detectar framework, identificar features, generar tests de API, generar tests E2E, ejecutar y crear resumen. Sin carpeta de sub‑pasos.

20–40 minutos.
Consultar también
No es sustituto de /bmad-code-review ni de /bmad-create-story en modo validate.
